# M0 — Bring-Up — Build Guide

> **How to use this guide:** You write 100% of the code by hand. Each task gives the
> **concept**, **why**, an ordered **how-to** (which Vulkan objects/calls, in what
> order — no code), **pitfalls**, and a **verification**. Work top to bottom. When a
> step is unclear or a validation error appears, ask — that is the guide working as
> intended. Checkboxes (`- [ ]`) are for tracking your own progress.

**Goal:** A Win32 window that brings up the full Vulkan device stack and clears the
screen to a solid color every frame, with validation layers clean.

**Architecture:** A thin hand-written Win32 platform layer behind `platform.hpp`; all
Vulkan code in `renderer_vk.cpp`; `volk` for function loading; Vulkan 1.3 **dynamic
rendering** (no render-pass/framebuffer objects).

**Tech Stack:** C++17/20 · Vulkan 1.3 · volk · Win32 · CMake

**Reference:** `docs/superpowers/specs/2026-05-17-vulkan-renderer-design.md`

---

## File Structure for M0

| File | Responsibility at end of M0 |
|------|----------------------------|
| `build.bat` | One-command build (configures + builds via CMake). |
| `CMakeLists.txt` | Compiles the src units, sets C++ standard, wires `volk`. |
| `.gitignore` | Excludes `build/`, IDE folders, compiled artifacts. |
| `third_party/volk/` | Vendored `volk.h` + `volk.c` (Vulkan function loader). |
| `src/main.cpp` | Entry point; owns the frame loop; ties platform + renderer. |
| `src/platform.hpp` | The OS-facing seam: window/input/time API + a surface hook. |
| `src/platform_win32.cpp` | Win32 implementation: window, message pump, timing. |
| `src/renderer.hpp` | Renderer public API: `renderer_init`, `renderer_render`, `renderer_shutdown`, `renderer_resize`. |
| `src/renderer_vk.cpp` | All Vulkan: instance, device, swapchain, frame loop, clear. |

`app.cpp` and `math.hpp` are **not** needed for M0 — they arrive at M2.

## Task 1: Project, Build & Repository Setup

**Files:** Create `build.bat`, `CMakeLists.txt`, `.gitignore`, `third_party/volk/`.

**Concept.** Before any rendering, you want a reproducible one-keystroke build and
version control. `volk` is a tiny loader that fetches Vulkan function pointers at
runtime, so you do not statically link the loader and you get faster dispatch.

**Why volk.** With `VK_NO_PROTOTYPES` defined, Vulkan functions are loaded by volk:
`volkInitialize()` once, `volkLoadInstance()` after the instance exists,
`volkLoadDevice()` after the device exists. This is the modern default and matters
later for extension functions.

- [ ] **Step 1 — Confirm the Vulkan SDK.** Verify the LunarG Vulkan SDK is installed
  and the `VULKAN_SDK` environment variable is set. You need its headers, the
  validation layers, and `glslc` (used from M1).
- [ ] **Step 2 — Vendor volk.** Download `volk.h` and `volk.c` into
  `third_party/volk/`. (volk itself only needs the Vulkan headers from the SDK.)
- [ ] **Step 3 — Write `CMakeLists.txt`.** Set CMake ≥ 3.20, C++ standard (17 or 20),
  the `RTGJ` executable from your `src/*.cpp` units plus `volk.c`. Add the SDK include
  path and `third_party/`. Define `VK_NO_PROTOTYPES` compile-wide. Do **not** link
  `vulkan-1.lib` — volk loads it. Define `WIN32_LEAN_AND_MEAN`.
- [ ] **Step 4 — Write `build.bat`.** It runs `cmake` configure into a `build/`
  directory then `cmake --build`. One command, no arguments.
- [ ] **Step 5 — Initialize git.** `git init`, write `.gitignore` (`build/`, `.vs/`,
  `*.user`, compiled SPIR-V if generated into the tree).
- [ ] **Step 6 — Verify.** Run `build.bat`. Expected: it compiles the current stub
  `main.cpp` and produces `RTGJ.exe` that runs and exits 0.
- [ ] **Step 7 — Commit.** `git add -A && git commit -m "chore: project + build setup"`

**Pitfalls:** linking the Vulkan loader *and* using `VK_NO_PROTOTYPES` causes symbol
conflicts — pick volk-only. CMake caching stale config — delete `build/` if configure
behaves oddly.

## Task 2: Win32 Window & Message Loop

**Files:** Create `src/platform.hpp`, `src/platform_win32.cpp`; update `src/main.cpp`.

**Concept.** The OS owns the window. You register a *window class* (which names a
*window procedure* callback), create a window from it, and continuously *pump*
messages the OS posts (input, resize, close).

**Why a seam.** `platform.hpp` is the only place above the OS layer. Nothing else
includes `<windows.h>`. The renderer will later ask this layer for a surface.

- [ ] **Step 1 — Design `platform.hpp`.** Declare a `PlatformWindow` (opaque/struct
  holding native handles, current width/height, a `should_close` flag, a
  `was_resized` flag) and functions: create the window, pump messages once, query
  high-resolution time. Keep native handle types behind `void*` or forward
  declarations so consumers need not include `<windows.h>`.
- [ ] **Step 2 — Register the window class.** In `platform_win32.cpp`, fill a
  `WNDCLASSEX`, point it at your *window procedure*, register it.
- [ ] **Step 3 — Create the window.** `CreateWindowEx` with a sane default size.
  Store `HWND` and `HINSTANCE` in the `PlatformWindow`.
- [ ] **Step 4 — Write the window procedure.** Handle `WM_CLOSE`/`WM_DESTROY` (set
  `should_close`), `WM_SIZE` (record new width/height, set `was_resized`). Forward
  everything else to `DefWindowProc`.
- [ ] **Step 5 — Write the message pump.** Use `PeekMessage` in a loop (non-blocking,
  unlike `GetMessage`) with `TranslateMessage` + `DispatchMessage`. A real-time
  renderer must not block waiting for messages.
- [ ] **Step 6 — Wire `main.cpp`.** Create the window, then loop: pump messages, until
  `should_close`.
- [ ] **Step 7 — Verify.** A window appears, can be moved and resized, and closing it
  exits the program cleanly.
- [ ] **Step 8 — Commit.** `git commit -m "feat: win32 window + message loop"`

**Pitfalls:** using `GetMessage` (blocks) instead of `PeekMessage`; forgetting
`WM_SIZE` so resize is invisible to your code; leaking the window class on exit (minor
at M0).

## Task 3: Vulkan Instance & Validation Layers

**Files:** Create `src/renderer.hpp`, `src/renderer_vk.cpp`.

**Concept.** A `VkInstance` is your connection to the Vulkan loader and tells it which
extensions and layers you want. **Validation layers** are a debug-only diagnostic
layer that reports API misuse with precise messages — your single most valuable tool.

**Why now.** Everything else needs the instance. Wiring validation first means every
later mistake is reported immediately.

- [ ] **Step 1 — Design `renderer.hpp`.** Declare `renderer_init` (takes the platform
  window), `renderer_render`, `renderer_resize`, `renderer_shutdown`. No Vulkan types
  in this header.
- [ ] **Step 2 — Initialize volk.** Call `volkInitialize()` first thing in
  `renderer_init`.
- [ ] **Step 3 — Build the instance.** Fill `VkApplicationInfo` with
  `apiVersion = VK_API_VERSION_1_3`. Enable instance extensions: `VK_KHR_surface`,
  `VK_KHR_win32_surface`, and (debug only) `VK_EXT_debug_utils`. Enable layer
  `VK_LAYER_KHRONOS_validation` in debug builds. Call `vkCreateInstance`.
- [ ] **Step 4 — Load instance functions.** Call `volkLoadInstance(instance)`.
- [ ] **Step 5 — Create the debug messenger.** Fill
  `VkDebugUtilsMessengerCreateInfoEXT` with a callback that prints severity + message.
  Create it with `vkCreateDebugUtilsMessengerEXT`. Optionally also pass the
  create-info via the instance's `pNext` to capture instance-creation messages.
- [ ] **Step 6 — Verify.** Program runs, no validation errors printed. Temporarily
  trigger one (e.g. a deliberately wrong call) to confirm the callback fires, then
  remove it.
- [ ] **Step 7 — Commit.** `git commit -m "feat: vulkan instance + validation"`

**Pitfalls:** enabling the validation layer when the SDK is absent → instance creation
fails; not checking layer/extension availability before requesting; forgetting
`volkLoadInstance` → instance-level functions are null.

## Task 4: Surface & Physical Device Selection

**Files:** Modify `src/renderer_vk.cpp`; add a surface hook to `platform_win32.cpp`.

**Concept.** A `VkSurfaceKHR` bridges Vulkan and your window. A *physical device* is a
GPU; you must pick one that can both render graphics and *present* to your surface.

- [ ] **Step 1 — Create the surface.** Add a function in the platform layer that, given
  the `HWND` + `HINSTANCE`, calls `vkCreateWin32SurfaceKHR` and returns a
  `VkSurfaceKHR`. This is the renderer↔platform surface hook from the spec.
- [ ] **Step 2 — Enumerate GPUs.** `vkEnumeratePhysicalDevices`.
- [ ] **Step 3 — Score each GPU.** For each: query queue family properties; find a
  family with `VK_QUEUE_GRAPHICS_BIT`; find a family that supports present
  (`vkGetPhysicalDeviceSurfaceSupportKHR`) — usually the same one; confirm the device
  advertises the `VK_KHR_swapchain` extension. Prefer a discrete GPU.
- [ ] **Step 4 — Store the choice.** Keep the chosen `VkPhysicalDevice` and the queue
  family index (or indices, if graphics and present differ).
- [ ] **Step 5 — Verify.** Log the chosen GPU name (`vkGetPhysicalDeviceProperties`);
  no validation errors.
- [ ] **Step 6 — Commit.** `git commit -m "feat: surface + physical device selection"`

**Pitfalls:** creating the surface before the instance enabled `VK_KHR_win32_surface`;
*assuming* graphics and present are the same family — handle both even though they
usually coincide on Windows desktop GPUs.

## Task 5: Logical Device & Queues

**Files:** Modify `src/renderer_vk.cpp`.

**Concept.** The *logical device* is your working handle to the GPU. A *queue* is
where you submit recorded work.

- [ ] **Step 1 — Describe the queue.** Fill `VkDeviceQueueCreateInfo` for your
  graphics(+present) family with one queue and a priority of 1.0. If present is a
  separate family, add a second create-info.
- [ ] **Step 2 — Create the device.** Request the `VK_KHR_swapchain` device extension.
  Enable the dynamic-rendering feature (core in 1.3 — request it via
  `VkPhysicalDeviceVulkan13Features.dynamicRendering`, chained through `pNext`). Call
  `vkCreateDevice`.
- [ ] **Step 3 — Load device functions.** Call `volkLoadDevice(device)`.
- [ ] **Step 4 — Retrieve the queue.** `vkGetDeviceQueue` for the family/index.
- [ ] **Step 5 — Verify.** Device and queue handles are non-null; no validation errors.
- [ ] **Step 6 — Commit.** `git commit -m "feat: logical device + queues"`

**Pitfalls:** forgetting `volkLoadDevice` → all device functions are null; not
requesting the swapchain extension; not enabling `dynamicRendering` (Task 8 needs it).

## Task 6: Swapchain & Image Views

**Files:** Modify `src/renderer_vk.cpp`.

**Concept.** The *swapchain* is a ring of images you render into and hand to the OS to
display. You negotiate format, size, and present mode with the surface. Each image
needs an *image view* to be used as a render target.

- [ ] **Step 1 — Query surface support.** Get capabilities
  (`vkGetPhysicalDeviceSurfaceCapabilitiesKHR`), formats, and present modes.
- [ ] **Step 2 — Choose the format.** Prefer `VK_FORMAT_B8G8R8A8_SRGB` with
  `SRGB_NONLINEAR` color space — an sRGB swapchain means the hardware does the final
  encode for you.
- [ ] **Step 3 — Choose the present mode.** `FIFO` is always available (vsync) — use
  it for M0. `MAILBOX` is the low-latency option for later.
- [ ] **Step 4 — Choose the extent.** On Windows the surface usually dictates
  `currentExtent`; the special `0xFFFFFFFF` value means you pick (clamp the window
  size to min/max).
- [ ] **Step 5 — Choose image count.** `minImageCount + 1`, clamped to `maxImageCount`
  (0 = no max).
- [ ] **Step 6 — Create the swapchain.** `vkCreateSwapchainKHR` with usage
  `COLOR_ATTACHMENT`. Retrieve images with `vkGetSwapchainImagesKHR`.
- [ ] **Step 7 — Create image views.** One `VkImageView` per swapchain image.
- [ ] **Step 8 — Verify.** N images + N views created; no validation errors.
- [ ] **Step 9 — Commit.** `git commit -m "feat: swapchain + image views"`

**Pitfalls:** ignoring the `currentExtent == 0xFFFFFFFF` case; a minimized window
reports extent (0,0) — you must not create a zero-sized swapchain (handled in Task 9).

## Task 7: Command Pool, Command Buffers & Sync Objects

**Files:** Modify `src/renderer_vk.cpp`.

**Concept.** Work is *recorded* into command buffers allocated from a *command pool*.
Ordering is enforced by *fences* (GPU→CPU) and *semaphores* (GPU→GPU). *Frames in
flight* let the CPU prepare the next frame while the GPU works on the current one.

- [ ] **Step 1 — Create the command pool.** For the graphics family, with the
  `RESET_COMMAND_BUFFER` flag so you can re-record each frame.
- [ ] **Step 2 — Pick FRAMES_IN_FLIGHT = 2.** Define a per-frame struct.
- [ ] **Step 3 — Allocate command buffers.** One primary command buffer per
  frame-in-flight.
- [ ] **Step 4 — Create sync objects.** Per frame-in-flight: an *image-available*
  semaphore, a *render-finished* semaphore, and an *in-flight fence* created in the
  **signaled** state.
- [ ] **Step 5 — Verify.** All objects created; no validation errors.
- [ ] **Step 6 — Commit.** `git commit -m "feat: command pool + sync objects"`

**Pitfalls:** creating the fence *unsignaled* → the first frame deadlocks waiting on
it. Note for later: a render-finished semaphore is ideally per-swapchain-image, not
per-frame-in-flight; per-frame is acceptable at M0 but revisit if you hit a present
sync warning.

## Task 8: Render Loop — Clear & Present

**Files:** Modify `src/renderer_vk.cpp` (`renderer_render`), call it from `main.cpp`.

**Concept.** Each frame: wait for this frame's fence, acquire a swapchain image,
record a clear, submit with the correct wait/signal semaphores, present.

**Decision — dynamic rendering vs render pass.** The spec's M1 note mentioned a render
pass; this guide refines that. **Use dynamic rendering** (`vkCmdBeginRendering`, core
1.3): no `VkRenderPass` or `VkFramebuffer` objects, you supply the color target and
clear value inline. It is simpler and M1 reuses it. If you later want to understand
the classic render-pass model, that is a worthwhile side study — but build M0 on
dynamic rendering.

- [ ] **Step 1 — Wait & reset the fence.** `vkWaitForFences` on this frame's fence,
  then `vkResetFences`.
- [ ] **Step 2 — Acquire an image.** `vkAcquireNextImageKHR`, signalling this frame's
  image-available semaphore. It returns a swapchain image index. Handle
  `OUT_OF_DATE`/`SUBOPTIMAL` (Task 9).
- [ ] **Step 3 — Record the command buffer.** Reset and begin it. Transition the
  acquired swapchain image from `UNDEFINED` to `COLOR_ATTACHMENT_OPTIMAL` with a
  pipeline barrier.
- [ ] **Step 4 — Begin rendering & clear.** `vkCmdBeginRendering` with the image's
  view as the color attachment, `loadOp = CLEAR`, a clear color, `storeOp = STORE`.
  Then `vkCmdEndRendering` immediately — the clear *is* the frame at M0.
- [ ] **Step 5 — Transition to present.** Barrier the image from
  `COLOR_ATTACHMENT_OPTIMAL` to `PRESENT_SRC_KHR`. End the command buffer.
- [ ] **Step 6 — Submit.** `vkQueueSubmit`: wait on the image-available semaphore at
  stage `COLOR_ATTACHMENT_OUTPUT`, signal the render-finished semaphore, pass this
  frame's fence.
- [ ] **Step 7 — Present.** `vkQueuePresentKHR`: wait on the render-finished
  semaphore, with the swapchain + image index.
- [ ] **Step 8 — Advance the frame index** (modulo FRAMES_IN_FLIGHT).
- [ ] **Step 9 — Verify.** The window shows your solid clear color, steadily, with no
  validation errors and no flicker. Capture one frame in **RenderDoc** to confirm the
  barriers and clear look clean.
- [ ] **Step 10 — Commit.** `git commit -m "feat: render loop clears the screen"`

**Pitfalls:** wrong wait stage on the submit; presenting before render-finished is
signalled; forgetting the final `PRESENT_SRC` transition (validation will flag it);
re-recording a command buffer still in use by the GPU because the fence wait was
skipped.

## Task 9: Swapchain Recreation on Resize

**Files:** Modify `src/renderer_vk.cpp`; add `renderer_resize`.

**Concept.** When the window resizes, the swapchain no longer matches the surface and
is *stale*. Vulkan signals this via `VK_ERROR_OUT_OF_DATE_KHR` (and sometimes
`SUBOPTIMAL`) from acquire/present; you must rebuild it.

- [ ] **Step 1 — Detect staleness.** Trigger recreation when acquire or present
  returns `OUT_OF_DATE`/`SUBOPTIMAL`, or when the platform's `was_resized` flag is set.
- [ ] **Step 2 — Drain the GPU.** `vkDeviceWaitIdle` before destroying anything —
  never destroy resources the GPU may still be using.
- [ ] **Step 3 — Destroy old swapchain resources.** Destroy the image views, then the
  swapchain.
- [ ] **Step 4 — Recreate.** Re-run the Task 6 path with the new surface extent.
- [ ] **Step 5 — Handle minimize.** If the extent is (0,0) (minimized), skip rendering
  entirely until it is non-zero — do not create a zero-sized swapchain.
- [ ] **Step 6 — Verify.** Resize and maximize/restore the window repeatedly: the
  clear color keeps filling the whole window, no validation errors, no leaks.
- [ ] **Step 7 — Commit.** `git commit -m "feat: swapchain recreation on resize"`

**Pitfalls:** destroying resources without `vkDeviceWaitIdle` first; leaking the old
image views; recreating mid-frame; busy-spinning while minimized.

## M0 Definition of Done

- [ ] `build.bat` builds clean; `RTGJ.exe` opens a window that clears to a color.
- [ ] Resize, maximize, minimize/restore all behave correctly.
- [ ] **Zero Vulkan validation errors or warnings** in a debug build.
- [ ] A RenderDoc capture shows a clean clear with correct image layout transitions.
- [ ] `renderer_shutdown` destroys everything in reverse creation order — no leaks
  reported on exit.

When all boxes are checked, M0 is complete and you are ready for the **M1 — First
Triangle** build guide.
