# Logger Module Implementation Plan

> **For the builder:** You write 100% of the code by hand. Each step gives the
> **how-to** (which spdlog objects/calls, in what order — no code), an exact command
> where one applies, and the **expected result**. Work top to bottom. Checkboxes
> (`- [ ]`) track your progress. When a step is unclear, ask — that is the plan
> working as intended.

**Goal:** A `core` logging module — `RTGJ_LOG_*` macros over spdlog — that prints
colored output to the console and writes a rotating log file, with trace/debug
stripped from Release builds.

**Architecture:** A macro-wrapper seam (`src/core/log.hpp`) over spdlog. The macros
forward to spdlog's `SPDLOG_*` macros; all sink/level/pattern configuration lives in
`src/core/log.cpp`, the only `.cpp` that includes spdlog directly. Compile-time level
stripping is driven by a per-build-config `SPDLOG_ACTIVE_LEVEL` compile definition.

**Tech Stack:** C++20 · spdlog v1.17.0 (compiled lib, already linked) · CMake +
Visual Studio 2026 multi-config generator.

**Spec:** `docs/superpowers/specs/2026-05-18-logger-design.md`

## Task 1: CMake — Per-Config Compile-Time Log Level

**Files:** Modify `CMakeLists.txt`.

**Concept.** `SPDLOG_ACTIVE_LEVEL` is a preprocessor value spdlog reads to decide
which `SPDLOG_*` macros compile to nothing. It must be a *compile definition* (set on
the command line, so it is defined before any include) and must differ between Debug
and Release. The Visual Studio generator is multi-config, so the value is chosen with
a generator expression, not an `if()`.

- [ ] **Step 1 — Add the compile definition.** In `CMakeLists.txt`, extend the
  existing `target_compile_definitions(${PROJECT_NAME} PRIVATE …)` line. Add
  `SPDLOG_ACTIVE_LEVEL` set, via a `$<IF:$<CONFIG:Debug>,…,…>` generator expression,
  to `SPDLOG_LEVEL_TRACE` in Debug and `SPDLOG_LEVEL_INFO` otherwise.
- [ ] **Step 2 — Re-configure.** Run: `cmake --preset windows-x64`
  Expected: configures with no errors (no compile yet — nothing references the
  definition).
- [ ] **Step 3 — Commit.** Run:
  `git add CMakeLists.txt`
  `git commit -m "build: add SPDLOG_ACTIVE_LEVEL per-config compile definition"`

**Pitfall.** Do not `#define SPDLOG_ACTIVE_LEVEL` inside a header — if it lands after
spdlog is included, stripping silently does nothing. The compile definition avoids
the ordering trap.

## Task 3: The Public Header — `log.hpp`

**Files:** Fill stub `src/core/log.hpp` (currently just `#pragma once`).

**Concept.** This header is the logger's entire public surface. It includes
`<spdlog/spdlog.h>` (unavoidable — the macros forward to spdlog's macros, which must
be visible where they expand), declares the two lifecycle functions, and defines the
six `RTGJ_LOG_*` macros.

- [ ] **Step 1 — Include spdlog.** Below `#pragma once`, include `<spdlog/spdlog.h>`.
  This is the one header that pulls spdlog in for the whole codebase.
- [ ] **Step 2 — Declare the lifecycle functions.** Declare `void log_init();` and
  `void log_shutdown();` — free functions, matching the `platform_*` style.
- [ ] **Step 3 — Define the six macros.** Define `RTGJ_LOG_TRACE`, `RTGJ_LOG_DEBUG`,
  `RTGJ_LOG_INFO`, `RTGJ_LOG_WARN`, `RTGJ_LOG_ERROR`, `RTGJ_LOG_CRITICAL`. Each is a
  variadic macro that forwards its arguments straight to the matching spdlog macro
  (`SPDLOG_TRACE`, `SPDLOG_DEBUG`, `SPDLOG_INFO`, `SPDLOG_WARN`, `SPDLOG_ERROR`,
  `SPDLOG_CRITICAL`). Forward with `__VA_ARGS__`. Forwarding to spdlog's macros (not
  its functions) is what gives you automatic `file:line` capture and
  `SPDLOG_ACTIVE_LEVEL` stripping for free.
- [ ] **Step 4 — Commit.** Run:
  `git add src/core/log.hpp`
  `git commit -m "feat: logger public header with RTGJ_LOG_* macros"`

**Pitfall.** Use `SPDLOG_TRACE` (targets the default logger), not
`SPDLOG_LOGGER_TRACE` (needs a logger argument) — Task 4 registers your logger as the
default so the no-argument form works.

## Task 4: The Configuration Site — `log.cpp`

**Files:** Fill stub `src/core/log.cpp` (currently empty).

**Concept.** `log_init` builds the logging stack once: two sinks → one logger → set
as spdlog's default → pattern, runtime level, flush policy. `log_shutdown` flushes
and releases it. This is the only `.cpp` that includes spdlog directly.

- [ ] **Step 1 — Includes.** Include `"core/log.hpp"`. Then include the sink headers
  you need: `<spdlog/sinks/stdout_color_sinks.h>` and
  `<spdlog/sinks/rotating_file_sink.h>`.
- [ ] **Step 2 — Create the console sink.** In `log_init`, construct a
  `stdout_color_sink_mt` (the `_mt` multi-threaded variant — the renderer gains
  worker threads later). Hold it in a `spdlog::sink_ptr`.
- [ ] **Step 3 — Create the rotating file sink.** Construct a
  `rotating_file_sink_mt` targeting `logs/rtgj.log`, max size 5 MB
  (`5 * 1024 * 1024`), 3 rotated files. spdlog creates the `logs/` directory itself.
- [ ] **Step 4 — Assemble the logger.** Create one `spdlog::logger` named `"rtgj"`
  holding both sinks (spdlog's logger constructor accepts a sink list). Wrap it in a
  `std::shared_ptr`.
- [ ] **Step 5 — Register as default.** Pass the logger to
  `spdlog::set_default_logger` so the `SPDLOG_*` macros target it.
- [ ] **Step 6 — Pattern.** Call `spdlog::set_pattern` with a pattern containing a
  timestamp, the colored level (`%^%l%$`), and source location (`%s:%#`), then `%v`
  for the message.
- [ ] **Step 7 — Runtime level.** Call `spdlog::set_level` — `trace` in Debug,
  `info` in Release. Gate this with `#ifdef NDEBUG` (or `#ifndef NDEBUG`); `NDEBUG`
  is defined by the compiler in Release configs.
- [ ] **Step 8 — Flush policy.** Call `spdlog::flush_on(spdlog::level::warn)` so a
  crash still leaves warnings and errors on disk.
- [ ] **Step 9 — Implement `log_shutdown`.** Call `spdlog::shutdown()` — it flushes
  and drops all registered loggers.
- [ ] **Step 10 — Commit.** Run:
  `git add src/core/log.cpp`
  `git commit -m "feat: logger sink + level configuration"`

**Pitfall.** If you skip `set_default_logger`, the macros log through spdlog's stock
default logger and you will see uncolored output with no file — a sign this step was
missed.

## Task 5: Wire the Lifecycle & Verify — `main.cpp`

**Files:** Modify `src/main.cpp`.

**Concept.** The logger must be alive before anything that might fail. `log_init`
runs first in `main`, `log_shutdown` runs last. A smoke-test line proves the whole
stack works end to end.

- [ ] **Step 1 — Include the header.** Add `#include "core/log.hpp"` alongside the
  existing `#include "platform/platform.hpp"`.
- [ ] **Step 2 — Call `log_init` first.** Make `log_init()` the first statement in
  `main`, before the `platform_init` call.
- [ ] **Step 3 — Add a smoke-test line.** After `log_init()`, add a
  `RTGJ_LOG_INFO("RTGJ starting up")` (or similar).
- [ ] **Step 4 — Call `log_shutdown` last.** Add `log_shutdown()` as the final
  statement before `return 0`, after `platform_shutdown()`.
- [ ] **Step 5 — Build.** Run: `cmake --build build --preset windows-x64-debug`
  Expected: compiles and links with no errors; produces `build/Debug/RTGJ.exe`.
- [ ] **Step 6 — Run and verify.** Run: `build/Debug/RTGJ.exe`
  Expected: a colored `[info]` line "RTGJ starting up" appears in the console, the
  window opens, and closing it exits cleanly.
- [ ] **Step 7 — Verify the file sink.** Confirm `logs/rtgj.log` now exists and
  contains the same startup line. (Open it with the Read tool or your editor.)
- [ ] **Step 8 — Commit.** Run:
  `git add src/main.cpp`
  `git commit -m "feat: initialize logger in main lifecycle"`

**Pitfall.** Any `RTGJ_LOG_*` call placed before `log_init` logs through spdlog's
stock default logger — keep `log_init` first.

## Task 7: Verify Release Stripping

**Files:** None — verification only.

**Concept.** The whole reason for the macro approach is that `RTGJ_LOG_TRACE` and
`RTGJ_LOG_DEBUG` vanish in Release. This task confirms it.

- [ ] **Step 1 — Add temporary probe lines.** In `main`, just after the smoke-test
  `INFO`, add a `RTGJ_LOG_TRACE(...)` and a `RTGJ_LOG_DEBUG(...)` with distinctive
  text.
- [ ] **Step 2 — Build Release.** Run:
  `cmake --build build --preset windows-x64-release`
  Expected: builds with no errors.
- [ ] **Step 3 — Run Release and verify.** Run: `build/Release/RTGJ.exe`
  Expected: the `INFO` line appears; the trace and debug lines do **not** — they were
  compiled out by `SPDLOG_ACTIVE_LEVEL=SPDLOG_LEVEL_INFO`.
- [ ] **Step 4 — Build & run Debug to confirm the contrast.** Run:
  `cmake --build build --preset windows-x64-debug` then `build/Debug/RTGJ.exe`
  Expected: all three lines (info, trace, debug) appear.
- [ ] **Step 5 — Remove the probe lines.** Delete the temporary `RTGJ_LOG_TRACE` /
  `RTGJ_LOG_DEBUG` from `main`.
- [ ] **Step 6 — Commit.** Run:
  `git add src/main.cpp`
  `git commit -m "test: verify release strips trace/debug logging"`

## Definition of Done

- [ ] `RTGJ.exe` prints a colored startup `INFO` line to the console.
- [ ] `logs/rtgj.log` is created and contains that line.
- [ ] A Release build emits no `trace`/`debug` output; a Debug build emits all levels.
- [ ] `platform_win32.cpp` no longer calls `fprintf`; failures go through
  `RTGJ_LOG_ERROR`.
- [ ] No spdlog header is `#include`d by any `.cpp` other than `log.cpp`.

When all boxes are checked, the logger is complete and you are ready to return to
**M0 Task 3 — Vulkan Instance & Validation Layers**, whose debug-messenger callback
will log through `RTGJ_LOG_*`.
